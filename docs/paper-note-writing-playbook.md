# 论文解读写作规范

本规范面向本仓库中的中文论文解读或评论文章，目标是产出可发布、可复用、对读者友好的长文，而不是临时笔记。

## 1. 总原则

1. 正文必须自洽。
   - 不写“这就是原文 Methods 式 (15) 的含义”这类句子。
   - 读者不翻原论文，也应该能看懂我们在说什么。
   - 引用原文和 SI 的作用是提供依据，不是承担解释链条。

2. 语气要客观、严肃、节制。
   - 不写营销式评价。
   - 不轻易说“彻底解决”“本质统一”等过强判断。
   - 所有优点都要对应具体结构或实验现象，所有局限都要指出约束条件。

3. 问题意识要先于公式堆砌。
   - 先回答“为什么需要这个方法”，再回答“这个方法怎么构造”。
   - 先区分背景、动机、方法、实现、实验、评述，再展开细节。

4. 文章应服务于网站读者，而不是复述论文目录。
   - 结构可以参考原文，但不能被原文节号牵着走。
   - 遇到原文标号混乱、记号跳跃、论证过紧凑时，要主动重组。

## 2. 推荐结构

推荐主线如下：

1. 导言
2. 前置知识
3. 研究问题与方法设计
4. 数学构造
5. 算法实现与系统复杂度
6. 实验结果与物理解释
7. 总结与评述
8. 附录

其中：

- 第 3 节讲“为什么这样设计”。
- 第 4 节讲“公式上如何成立”。
- 第 5 节讲“实现时具体怎么做、复杂度为什么成立”。
- 第 6 节讲“实验到底证明了什么，边界在哪里”。
- 第 7 节必须有独立判断，不能只是摘要复述。

## 3. 数学与公式写法

1. 直接写出关键公式，不要只写“对应原文某式”。
2. 向量统一用 `\mathbf{}` 粗体。
3. 所有容易混淆的量都要补定义：
   - 维度
   - 索引含义
   - 张量切片方式
   - 是否为复数、是否需要共轭
4. 推导尽量完整，至少要补足原文中默认省略的中间步骤。
5. 只要公式里出现新符号，就在首次出现处解释，不把读者留到后文。
6. 需要收缩、广播或张量积时，优先用爱因斯坦求和约定或明确维度说明，避免“矩阵式直觉描述”。

## 4. 图与图注规范

1. 优先使用论文源码图，其次才是 PDF 裁剪图。
2. 图必须直接支撑当前段落，不放“装饰性引用图”。
3. 图名统一、可维护。
   - 当前仓库建议按章节命名，例如 `s06-cumulene.png`。
4. 同一篇文章的图片放在独立目录，保持目录整洁。
5. 图注要说明这张图证明了什么，而不只是描述图中有什么。

## 5. 实验部分写法

每个实验小节尽量按以下顺序组织：

1. 实验背景：这个任务为什么重要，针对哪类失效模式。
2. 配置：数据、体系、cutoff、层数、对比模型、关键超参数。
3. 方法：到底比较了哪些方案。
4. 结果：结论是什么。
5. 解释：这个结果支持了什么判断，不支持什么判断。

要求：

- 简明，但不能把关键设定省掉。
- 结果要落到具体现象，不只给“更好/更差”。
- 明确区分“证明必要性”的实验与“补充说明”的实验。

## 6. 评述部分写法

第 7 节至少应回答以下问题：

1. 这篇文章真正的贡献是什么。
2. 它解决的是哪一个已有结构性矛盾。
3. 它和相邻路线是什么关系：
   - 标准 attention
   - 局部等变 MPNN
   - 物理启发的长程修正
   - 频域/平面波/Ewald 路线
4. 它的局限是什么。
5. 哪些问题仍然是开放的。

评述要避免两个极端：

- 只复述原文结论，没有自己的判断。
- 评价过度，用没有证据支撑的强断言。

## 7. 附录取舍原则

附录只放两类内容：

1. 不影响主线，但保留后能显著降低后续维护成本的推导或对比。
2. 符号表、实现约定、与相关方法的形式关系。

凡是已经在正文完整讲清楚的内容，不再在附录重复。

## 8. 本站特定约定

1. front matter 中：
   - `description` 保持纯文本摘要。
   - DOI 链接放在正文导言，而不是把 Markdown 链接塞进 `description`。
   - `tags` 用栏目级主题。
   - `keywords` 用论文专有术语。

2. 图片：
   - 用 Markdown 图片语法插入。
   - 图题通过图片 title 渲染为 `figcaption`。

3. 公式：
   - 桌面端保留行间公式编号。
   - 移动端避免编号与公式本体重叠，必要时把编号移到公式下方。

## 9. 每次发布前的检查清单

- 标题、摘要、DOI、日期是否正确
- `draft` 是否符合发布状态
- `tags` 和 `keywords` 是否已填写
- 公式编号是否连续、是否和正文引用一致
- 新符号是否首次定义
- 图是否都来自当前实际使用版本
- 图片命名和目录是否干净
- 移动端公式是否与编号或正文重叠
- `npm run build` 是否通过

## 10. 针对 EFA 这篇文章得到的额外经验

1. 对几何深度学习论文，最容易写坏的地方不是公式本身，而是“问题张力”没有讲清楚。
2. 对 attention 论文，必须区分：
   - attention score 在表达什么
   - output structure 在表达什么
   - invariance / equivariance 是在什么层面满足的
3. 对实现节，维度说明往往比公式本身更重要。
4. 对实验节，最好把“长程”“全局”“方向敏感”“电子离域”这几类能力分开写，不要混成一个笼统优点。
